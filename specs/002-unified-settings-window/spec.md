# Feature Specification: See See 统一设置主窗口

**Feature Branch**: `master`

**Created**: 2026-07-23

**Status**: Approved

**Input**: User description: "将主窗口重构为 Windows Raycast 风格的统一设置窗口，把常规、模型、提示词、历史和关于合并到一个窗口；主窗口不提供截图入口，保留全局快捷键并在托盘菜单提供开始截图。"

## User Scenarios & Testing

### User Story 1 - 在一个窗口管理全部设置 (Priority: P1)

用户打开 See See 后，通过固定侧栏在常规、模型、提示词、历史和关于之间切换，不再管理多个重复窗口。

**Why this priority**: 统一导航和设置窗口是本次重构的核心价值，也是消除当前割裂体验的前提。

**Independent Test**: 启动应用，依次切换五个栏目并完成一次模型编辑、提示词编辑和历史查看；始终只有主窗口参与管理。

**Acceptance Scenarios**:

1. **Given** 应用已启动，**When** 用户打开 See See，**Then** 主窗口默认展示常规栏目且不存在窗口内截图按钮。
2. **Given** 主窗口可见，**When** 用户切换侧栏栏目，**Then** 右侧只显示对应内容且不会创建额外窗口。
3. **Given** 首次设置尚未完成，**When** 用户选择配置模型或管理提示词，**Then** 当前主窗口切换到对应栏目。
4. **Given** 主窗口被关闭，**When** 应用仍在后台运行，**Then** 再次打开只显示并聚焦同一个主窗口。

---

### User Story 2 - 从快捷键或托盘开始截图 (Priority: P1)

用户通过已配置的全局快捷键或托盘菜单中的“开始截图”触发截图，不需要在设置窗口寻找截图按钮。

**Why this priority**: 截图仍是产品核心任务，入口迁移不能破坏现有快捷流程。

**Independent Test**: 分别使用全局快捷键和托盘菜单触发截图，验证两者进入同一截图流程。

**Acceptance Scenarios**:

1. **Given** 应用在后台运行，**When** 用户按全局快捷键，**Then** 应用按现有规则启动截图。
2. **Given** 托盘菜单已打开，**When** 用户选择“开始截图”，**Then** 应用复用同一截图流程。
3. **Given** 用户打开托盘菜单，**Then** 可见“开始截图”“打开 See See”和“退出”，不再出现独立历史或设置窗口入口。

---

### User Story 3 - 使用一致且可访问的设置界面 (Priority: P2)

用户在明暗主题、不同窗口尺寸和键盘操作下获得一致的 Windows Raycast 风格界面，危险操作使用应用内确认弹窗。

**Why this priority**: 视觉质量、响应式和无障碍是“完全重构”能够交付而不是只换颜色的验收条件。

**Independent Test**: 在代表性尺寸和系统主题下完成键盘导航、设置修改、危险操作确认与取消。

**Acceptance Scenarios**:

1. **Given** 系统主题变化，**When** 主窗口仍打开，**Then** 界面自动应用对应主题且无需重新启动。
2. **Given** 用户只使用键盘，**When** 浏览侧栏和设置项，**Then** 焦点顺序与视觉顺序一致且焦点清晰。
3. **Given** 用户执行删除或清空操作，**When** 确认弹窗打开，**Then** 默认聚焦取消，Escape 取消，关闭后焦点返回触发按钮。
4. **Given** 窗口宽度较小或文字缩放为 200%，**When** 用户浏览设置，**Then** 栏目和关键操作仍可到达且无水平内容丢失。

### Edge Cases

- 首次设置期间从模型或提示词返回常规栏目时，必须重新读取最新就绪状态。
- 托盘截图在已有截图会话或分析任务运行时，必须沿用当前的冲突处理，不创建第二条流程。
- 未知窗口标签不得被误认为管理窗口；捕获和结果窗口继续使用专用视图。
- 主窗口隐藏后再次显示必须保留可用状态，不得残留已销毁的独立窗口引用。
- 原生确认弹窗不支持或测试环境缺少 `showModal` 时，组件仍必须保持可测试的开放状态和语义。

## Requirements

### Functional Requirements

- **FR-001**: 系统必须使用唯一的主窗口承载常规、模型、提示词、历史和关于栏目。
- **FR-002**: 系统不得从主窗口提供截图按钮或截图命令入口。
- **FR-003**: 系统必须保留现有全局截图快捷键及其配置能力。
- **FR-004**: 托盘或菜单栏必须提供一个“开始截图”命令并复用现有截图流程。
- **FR-005**: 托盘必须继续提供打开主窗口和退出应用的操作。
- **FR-006**: 系统必须停止创建独立的设置、提示词和历史窗口。
- **FR-007**: 首次设置必须嵌入常规栏目，并能在同一窗口切换到模型或提示词栏目。
- **FR-008**: 模型、提示词、历史、截图、结果和本地设置的现有业务行为必须保持不变。
- **FR-009**: 危险操作必须使用统一的应用内确认弹窗，不得继续调用浏览器系统确认框。
- **FR-010**: 关于栏目必须展示产品说明、运行版本和本地隐私说明。
- **FR-011**: 应用版本必须同步更新为 0.2.0。

### User Experience and UI Requirements

- **UX-001**: 常规、模型、提示词、历史和关于必须使用同一侧栏导航、标题层级和反馈样式。
- **UX-002**: 加载、空白、成功、错误、禁用和恢复状态必须继续可见且可操作。
- **UI-001**: 界面必须复用集中设计 token、现有表单组件和少量共享图标/确认弹窗，不得新增 UI 框架。
- **UI-002**: 所有交互必须支持键盘，使用可见 `:focus-visible`，并尊重系统明暗主题与减少动态效果设置。
- **UI-003**: 视觉验收必须覆盖 1024×720、1440×900、200% 文字缩放，以及 360×240 结果浮窗。
- **UI-004**: 主窗口必须保留原生标题栏，不使用透明窗口、背景模糊、Mica、Acrylic、Vibrancy 或自绘窗口按钮。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户能在一次主窗口会话内访问全部五个管理栏目，不产生额外管理窗口。
- **SC-002**: 全局快捷键和托盘“开始截图”两条路径均能在一次操作内进入现有截图流程。
- **SC-003**: 1024×720、1440×900 和 200% 文字缩放下，所有关键设置与操作均可通过键盘到达且无水平内容丢失。
- **SC-004**: 所有危险操作的确认、取消、Escape 和焦点恢复场景均通过自动化测试。
- **SC-005**: 现有前端、E2E 和 Rust 测试全部通过，且发布构建可安装并启动。

## Assumptions

- “主窗口不提供截图入口”不影响全局快捷键。
- 托盘中只有一个截图命令；打开应用和退出属于生命周期操作，不算截图入口。
- Windows 参考图决定信息布局和视觉语言，但不会复制账号、云同步、主题工作室或设置搜索。
- 现有本地数据、凭据和历史不需要迁移。
